# RX Team — ASG Bot + Site — ТЗ (фінал)

Telegram-бот + сайт (**rxteam.pl**) для організації airsoft/ASG ігор спільноти **RX Team (Wrocław)**.
Неформальна група друзів. Проєкт замінює ручні процеси і поточну автоматизацію на Make.

> Технічні рішення — на виконавцеві. Бізнес-логіка нижче — узгоджена з організатором (5 блоків Q&A).
> **Заглушки** (значення, які організатор замінить пізніше) позначені `[заглушка]`.

---

## 1. Стек (рішення виконавця)

- **Бот:** grammY (TypeScript), webhook на Vercel
- **Сайт:** Next.js на Vercel, нейтральний стиль (фінальний стиль організатор зробить у Google AI Studio і передасть для натягування)
- **База + авторизація:** Supabase (Postgres + Auth)
- **Планувальник:** Vercel Cron / Supabase pg_cron
- **Домен:** rxteam.pl · **Мови скрізь:** PL / EN / UA (без RU)
- Масштаб ~30 гравців/гра, 376 у групі → безкоштовні тарифи

## 2. Принципи

- **Усе максимально в налаштування** (керує супер-адмін): ціни, кількості, бали, вартість звань, ролі, локації, тексти, таймінги.
- **Перемикач on/off на кожну значущу функцію** (feature toggles) в адмінці.
- Один профіль `player` = одна людина; вхід через Telegram **або** email, з можливістю прив'язати обидва → історія єдина.

## 3. Ролі та права

- **Майстер-адмін:** @delltex. Роздає ролі, керує адмінами, єдиний має доступ до **Налаштувань**.
- **Адмінів:** 5. Майстер призначає кожному набір прав:
  - **Ігри** — ігри, локації, анонси, голосування
  - **Оренда** — заявки, відмітка «комплект зайнятий/оплачений», запас комплектів *(напр. @BaRon25cm)*
  - **Чек-ін / явка** — ручний чек-ін, неявки, прощення штрафів
  - **Реферали** — підтвердження рефералів (з фото)
  - **Гравці** — зміна позивних, профілі, коригування балів
  - **Заявки в групу** — ручні рішення по шилду
- Звичайний гравець — без адмін-прав.

## 4. Анти-бот шилд (Етап 1, будуємо першим)

- `chat_join_request` → бот DM капчу через `user_chat_id` (5-хв вікно, без /start)
- Капча **для всіх**, мовами PL/EN/UA
- Правильно → `approveChatJoinRequest`; невірно/тиша/таймаут → `declineChatJoinRequest` (можна подати заявку знову)
- Без «білого списку»
- Перемикач функції в адмінці

## 5. Профіль та ідентичність

- Поля профілю: **ім'я, позивний, TG-нік**. (Телефон збираємо для верифікації рефералів/транспорту, але це не публічне поле профілю.)
- **Позивний** — вписується при першій реєстрації, **унікальний**, змінює лише адмін (право «Гравці»)
- Публічно (рейтинг): позивний + звання + «зароблено всього». Повний профіль/історія — після входу. *(припущення; уточнити за потреби)*
- `identities`: provider (telegram|email), external_id, verified. Реєстрація на сайті — **відкрита всім** (підтвердження пошти).

## 6. Ігри, реєстрація, чек-ін

- Ігри **нерегулярні** (бувають додаткові, буває без ігор). Запис можливий на **будь-яку анонсовану** гру; на кілька ігор одночасно — можна.
- Реєстрація відкрита **до −9 год**; скасування без штрафу **до −24 год**; запис після −24 год = зобов'язання.
- **Чек-ін (геолокація):** вікно = старт **±1 год** (старт 09:00 → 08:00–10:00). Координати з локації гри (полігони різні). TG — кнопка локації; сайт (email-юзери) — геолокація браузера. Перевірка: радіус + час.
- **Реєстрація + чек-ін = +1 гра.**
- **Ручний чек-ін** адміном — є. **Прощення штрафу** за неявку адміном — є.
- **Неявка:** registered + без скасування + без чек-іну → штраф балами (фонове завдання після вікна чек-іну).

## 7. Локації, анонси, голосування

- Полігони — збережені об'єкти (назва, координати, радіус, карта, фото).
- Анонс = шаблон + локація + дата → бот авто-постить у топік «Анонси».
- **Голосування за локацію** за −14 днів: бот постить опитування зі списку полігонів → грати там, де хочуть гравці (ініціює адмін під заплановану гру).

## 8. Оренда

- **Повні комплекти.** Запас — налаштування `[заглушка: 5]` (докуповуються).
- Потік: гравець у боті тисне «потрібна оренда» → бот показує **картку оренди** (фото + ціна + опис, з налаштувань) → заявка летить орендному адміну → адмін зв'язується, бере **передоплату** → відмічає «комплект зайнятий/оплачений».
- **Оренда платна** `[заглушка: 160–170 zł за типом]` — окремо від складки; на сайті — окремий блок (платна послуга, не змішувати з добровільною складкою).
- Перемикач функції в адмінці.

## 9. Транспорт (carpooling)

- Питання при реєстрації **обов'язкове**: своїм ходом / потребую.
- Своїм ходом → **звідки (вільний текст)** + кількість вільних місць.
- Потребую → бачу список водіїв (хто і звідки), пишу їм сам.
- Водій керує сам: ставить місця, **закриває** набір коли фул, відкриває заново, міняє к-сть. Складної логіки бронювання не будуємо.

## 10. Реферали

- Особисте посилання. Запрошувати може **тільки той, хто сам уже грав** (≥1 чек-ін).
- Знижка діє **на ту саму гру**, куди приведено новачків (не переноситься):
  - 1 новачок із чек-іном → **−50%** складки; 2+ → **безкоштовно**.
- Зараховується **з першої гри** новачка, після його чек-іну. Новачок нічого не отримує.
- Бали за друга — **як за гру** (`+10 [заглушка]`), теж з першої гри після чек-іну.
- Видно і адміну (екран чек-іну), і самому гравцю на телефоні («підтверджено −50% / безкоштовно») — показує при оплаті.
- **Захист від абʼюзу:** право запрошувати лише в тих, хто грав + підтвердження адміном **з фото** новачка + малий приз.

## 11. Економіка: бали, патч, звання

**Дві цифри балів:**
- **«Зароблено всього»** — сума за весь час, ніколи не падає → позиція в **таблиці рейтингу** (звідси «назавжди»).
- **«Доступний баланс»** — гаманець; росте від заробітку, **зменшується** при покупці звань/бонусів.

**Базові бали** `[заглушки]`: явка **+10**, неявка **−5** (пів явки), друг **+10**, ачівка **+5/+10/+20** за складністю.

**Патч (єдина нашивка RX Team = членство):**
- платний `[заглушка: ціна в налаштуваннях]`; **заявка на патч → адмін підтверджує → видача на грі**.
- **без патча:** бали капають на **15% менше (85%)**, накопичуються, але звання купити не можна; коли балів вистачає на наступне звання — підказка «потрібен патч».
- **з патчем:** 100% балів; вхідне звання **Recruit**, далі купуєш вище. Накопичені до патча бали — **можна** пускати на звання.

**Звання** (купуються за бали з доступного балансу, лише з патчем): **Recruit → Scout → Squad Leader → Team Leader**. Вартість у балах `[заглушки, налаштування]`. Бонуси кожного звання — `[пропишемо пізніше]`. Зіграно-ігор — лише статистика (і для ачівок).

**Магазин балів:** механіку закладаємо; перелік бонусів — `[пропишемо пізніше]`.

## 12. Ачівки, надійність, сезони

- **Reliability %** = явки/(явки+неявки), **за весь час**.
- **Ачівки** (мілітарі-назви, бали за складністю): First Contact (перша гра) · 10/25/50 Deployments · Iron Discipline (сезон без неявок) · Recruiter (перший друг) · Dawn Patrol (чек-ін у перші 10 хв). Кожна має перемикач/налаштування.
- **MVP — поки не робимо.**
- **Сезон = календарний квартал.** Бали **не обнуляються** (lifetime). Квартал використовується для **лотереї** і **«У цифрах»**.
- **Лотерея надійних:** 1 приз/квартал серед тих, у кого **0 неявок** за квартал.
- **«У цифрах»:** поквартально + річний підсумок.

## 18. Схема БД (чернетка)

- `players` — ім'я, позивний (unique), tg_nick, phone, games_played, points_earned, points_balance, has_patch, rank
- `identities` — player_id, provider (telegram|email), external_id, verified
- `admins` — player_id, permissions[] · `settings` — key/value (фіче-флаги + параметри)
- `locations` — назва, lat, lng, radius_m, map_url, photo
- `games` — location_id, start_at, reg_closes_at (−9год), cancel_deadline (−24год), checkin window (±1год), status
- `registrations` — game_id, player_id, status (registered|cancelled|no_show), needs_rental, transport (own|need), from_place, free_seats, seats_closed
- `checkins` — game_id, player_id, lat, lng, distance_m, source (tg|web), is_manual, created_at
- `referrals` — inviter_id, invited_id, game_id, status (pending|confirmed|rejected), photo
- `rentals` — game_id, player_id, status (requested|paid|occupied) · `rental_stock` (в settings)
- `patch_requests` — player_id, status (requested|approved|handed)
- `ranks` (config), `rank_purchases` — player_id, rank, cost, at
- `point_log` — player_id, delta, reason, game_id
- `achievements` (config), `player_achievements`
- `polls` — game_id/дата, варіанти-локації, голоси
- `shop_items` (config, пізніше), `purchases`

## 21. Етап 6 — Сайт (детальний план)

**Що вже є:** Next.js 14 App Router (лише `app/api/*` — бот і крони, фронтенд-сторінок НЕМАЄ). `lib/*` — уся доменна логіка (бали, сезон, ігри, реферали) перевикористовна на сервері сайту. Supabase — `@supabase/supabase-js` із **secret key, лише сервер**. Стиль — нейтральний (фінальний від організатора пізніше, Етап 7).

**Архітектурні рішення — ЗАФІКСОВАНО на 6.0:**
- **Auth:** Supabase Auth email **+ пароль** з підтвердженням пошти; **плюс вхід через Telegram Login Widget** (`/auth/telegram` → перевірка підпису → `ensurePlayer` → власна підписана сесія-cookie `rx_tg_session`, бо Supabase не знає TG-юзерів). Усі записи — через Server Actions/Route Handlers, що перевіряють сесію.
- **RLS лишається ВИМКНЕНО** — усі мутації через серверні дії з перевіркою ролі (відповідає поточному патерну бота на service-key).
