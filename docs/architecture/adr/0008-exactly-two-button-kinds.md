# 0008. Рівно два типи кнопок: action і delete

- **Статус:** Прийнято (Accepted) — **доповнено** [ADR-0026](0026-icon-overlay-fab-and-modal.md) (2026-06-16): додано 3-й тип `ghost` для нейтральних дій (cancel/attach). Поділ `action` ↔ `delete` лишається чинним.
- **Дата:** 2026-06-16
- **Категорія:** Дизайн-система
- **Теги:** buttons, design-system, ui-tokens, consistency

## Контекст і проблема
Розростання стилів кнопок (primary / secondary / ghost / outline у поєднанні з розмірними варіантами sm / md) фрагментує інтерфейс і робить вигляд UI неузгодженим між сторінками. Без чіткого обмеження кожен автор схильний вводити власний варіант кнопки під конкретний випадок. Команді потрібен мінімальний, передбачуваний словник кнопок, який неможливо «розмити» з часом.

## Розглянуті варіанти
- Класичний набір primary / secondary / ghost / outline — відхилено: породжує комбінаторний вибух стилів і візуальний дрейф.
- Збереження розмірних варіантів (sm / md / lg) — відхилено: висота й типографіка мають бути єдиними, варіювати треба лише ширину під довжину тексту.
- Довільне хардкодення Tailwind-класів кнопок по сторінках — відхилено: унеможливлює платформовий стандарт і єдине джерело правди.

## Рішення
Існує рівно ДВА типи кнопок — `action` (хакі, той самий колір що й заголовки, для create / save / confirm та будь-якої позитивної дії) і `delete` (червоний `--c-danger-solid`, для delete / cancel). Усі кнопки мають однакові шрифт, розмір шрифту і висоту; варіюється лише ШИРИНА (залежно від довжини тексту). Жодних secondary / ghost / outline і жодних розмірних варіантів (sm / md) не існує. Кнопки створюються виключно через `btn(kind)` або `<Button kind>`.

## Наслідки
- ✅ Візуальна узгодженість кнопок по всьому сайту — лише два передбачувані стани, ідентичні за висотою й типографікою.
- ✅ Єдине джерело правди: будь-яка зміна вигляду кнопки робиться в одному місці (`buttons.ts`), а не по сторінках.
- ✅ Менший когнітивний поріг для авторів — немає вибору між десятком варіантів, лише `action` чи `delete`.
- ⚠️ Будь-яка майбутня потреба у «третьому» вигляді кнопки (наприклад, нейтральна/другорядна дія) вимагає окремого перегляду цього ADR, а не локального обхідного рішення.
- ⚠️ Кнопки не можна стилізувати ad-hoc Tailwind-класами — усе має проходити через `btn()` / `<Button>`, що накладає дисципліну на код-рев'ю.
