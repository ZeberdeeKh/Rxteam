# 0010. Стилі лише через токени @/components/ui

- **Статус:** Прийнято (Accepted)
- **Дата:** 2026-06-16
- **Категорія:** Дизайн-система
- **Теги:** design-system, single-source-of-truth, tokens, tailwind

## Контекст і проблема
Дублювання Tailwind-класів для кнопок, поверхонь, заголовків і плашок по окремих сторінках призводить до візуального дрейфу. Коли кожна сторінка хардкодить власні класи, неможливо забезпечити єдиний стандарт у межах усієї платформи. Потрібне одне джерело правди для стилів, аналогічне підходу «single source of truth» у БД.

## Розглянуті варіанти
- **Хардкодити Tailwind-класи безпосередньо на сторінках** — відхилено: спричиняє розповзання стилів і робить централізований стандарт нездійсненним.
- **Підключити сторонню UI-бібліотеку (MUI/Radix/shadcn)** — відхилено: знімає контроль над дизайном і виходить за межі власної токен-системи.
- **Розкидані хелпери без єдиного бар'єльного експорту** — відхилено: важко знайти й перевикористати, тож дублювання все одно повертається.

## Рішення
Усе оформлення UI має походити лише з `@/components/ui` — токени `ui.*`, `btn()`, `badgeClass()`, `Collapsible` та nav-хелпери. Не хардкодити Tailwind-класи для кнопок, поверхонь, заголовків і плашок на сторінках. Файл `components/ui/styles.ts` оголошено ЄДИНИМ джерелом стилів.

## Наслідки
- ✅ Єдине джерело правди для стилів: зміна токена застосовується одразу по всьому сайту.
- ✅ Узгоджений вигляд кнопок, поверхонь, заголовків і плашок без візуального дрейфу між сторінками.
- ✅ Бар'єльний реекспорт через `components/ui/index.ts` спрощує імпорт і перевикористання.
- ⚠️ Будь-який новий патерн оформлення спершу треба додати до токен-системи, а не «швидко» на сторінці.
- ⚠️ Розробники мають знати токени `@/components/ui`; прямі Tailwind-класи в сторінках вважаються порушенням стандарту.
