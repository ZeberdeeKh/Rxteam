# 0017. Доступ до Supabase лише через lib/*

- **Статус:** Прийнято (Accepted)
- **Дата:** 2026-06-16
- **Категорія:** Архітектура
- **Теги:** supabase, data-layer, rls, security

## Контекст і проблема
У проєкті співіснують два режими доступу до Supabase: серверний клієнт на secret-key, який обходить RLS, і SSR-клієнт сесії. Якщо ці клієнти потраплять у браузер або будуть викликані ad-hoc прямо з компонентів, виникає ризик витоку секретного ключа й обходу авторизації. До того ж доменна логіка читання/запису легко дублюється між сайтом і ботом, якщо немає єдиного шару даних.

## Розглянуті варіанти
- Створювати Supabase-клієнт безпосередньо в компонентах за потреби — відкидаємо: ризик витоку secret-key у браузер і неконтрольований обхід RLS.
- Один універсальний клієнт на всі випадки (сервер + SSR + браузер) — відкидаємо: змішує привілеї та межі безпеки.
- Дублювати доменні запити окремо для сайту й для бота — відкидаємо: розсинхронізація логіки та подвійна підтримка.

## Рішення
Увесь доступ до БД/Supabase здійснюється виключно через модулі `lib/*`, ніколи напряму з компонентів. `lib/supabase.ts` містить клієнт на secret-key (server-only, обходить RLS, `persistSession` вимкнено); `lib/supabase-server.ts` — це SSR-клієнт сесії; `lib/supabase-browser.ts` — браузерний клієнт. Доменні читання/записи загорнуті в хелпери `lib/*` (site-data, admin-data, games, economy, settings тощо) і переused (не дублюються) спільно з ботом.

## Наслідки
- ✅ Секретний ключ ніколи не потрапляє в браузер: привілейований клієнт ізольовано в серверному коді з `persistSession:false`.
- ✅ Єдиний шар даних (`lib/*`) — доменна логіка переиспользується сайтом і ботом без дублювання.
- ✅ Чітке розмежування меж безпеки: secret-key, SSR-сесія та браузер — три окремі клієнти з різними привілеями.
- ⚠️ Будь-яке нове читання/запис мусить спершу отримати хелпер у `lib/*` — не можна звертатися до Supabase прямо з компонента.
- ⚠️ Оскільки RLS обходиться secret-key-клієнтом, авторизацію треба завжди забезпечувати на рівні застосунку (серверні дії + перевірка ролей), а не покладатися на БД.
