# 0018. RLS вимкнено — авторизація на сервері

- **Статус:** Прийнято (Accepted)
- **Дата:** 2026-06-16
- **Категорія:** Архітектура
- **Теги:** authz, rls-off, server-actions, rbac

## Контекст і проблема
Row Level Security у Supabase свідомо вимкнено, тож обмеження доступу неможливо забезпечити на рівні бази даних. Через це авторизацію доводиться повністю переносити в код застосунку. Потрібен послідовний, централізований механізм перевірки сесії та ролі для кожної мутації, щоб відсутність RLS не перетворилася на дірку в безпеці.

## Розглянуті варіанти
- **Увімкнути RLS і покладатися на політики БД** — відхилено: команда обрала тримати RLS вимкненим за дизайном і керувати доступом у застосунку.
- **Дозволяти прямі записи з браузера** — відхилено: без RLS це означало б неконтрольований доступ; усі записи мають проходити через сервер.
- **Розкидані ad-hoc перевірки ролі по сторінках** — відхилено на користь централізованого мапінгу сесії та іменованих дозволів.

## Рішення
RLS вимкнено за дизайном. Кожна мутація виконується на сервері (Server Action / Route Handler) з явними перевірками сесії та ролі. Мапінг «сесія → гравець» централізовано в `getSessionContext()` / `getSessionPlayer()`. Авторизація адміна — на основі полів (`players.is_master` / `is_admin` / `admin_perms[]`) з іменованими дозволами (`games|rental|checkin|referrals|players|joins|gallery`); сторінки гейтяться через `requirePerm()` / `requireMaster()`, повертаючи 404 не-адмінам, а оболонка адмінки закриває всю секцію.

## Наслідки
- ✅ Уся логіка доступу зосереджена в коді й проходить через одну точку (`getSessionContext()`), що робить її передбачуваною та аудитованою.
- ✅ Дрібнозерниста авторизація через іменовані дозволи дозволяє гнучко відкривати окремі розділи адмінки конкретним ролям.
- ✅ Повернення 404 (а не 403) не-адмінам приховує саме існування захищених сторінок.
- ⚠️ Безпека повністю залежить від дисципліни розробника: будь-яка мутація без явної перевірки сесії/ролі — це незахищений шлях, бо БД більше не страхує.
- ⚠️ Кожен новий запис обов'язково має йти через Server Action / Route Handler; прямі записи з браузера заборонені й мають вважатися порушенням стандарту.
