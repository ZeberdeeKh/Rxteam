# 0016. Next.js App Router + Server Actions

- **Статус:** Прийнято (Accepted)
- **Дата:** 2026-06-16
- **Категорія:** Архітектура
- **Теги:** nextjs, app-router, server-actions, ssr

## Контекст і проблема

Сайт спільноти має віддавати публічні дані через SSR заради SEO та швидкого першого рендеру, і водночас потребує безпечного шляху для записів. Проєкт побудований на Next.js 14 (App Router) і розгортається на Vercel. Потрібно чітко розділити читання й мутації так, щоб жоден запис не йшов напряму з браузера, а вся серверна логіка (включно з визначенням мови) виконувалася на сервері.

## Розглянуті варіанти

- **Класичний Pages Router з API-роутами для всього** — більше шаблонного коду, читання не отримує переваг Server Components, важче колокувати мутації поряд зі сторінками.
- **SPA з клієнтським фетчингом і прямими записами в БД з браузера** — ламає SEO/SSR і відкриває небезпечний шлях запису повз серверну авторизацію.
- **App Router лише для читання, а записи через зовнішній окремий бекенд-сервіс** — зайва інфраструктура та дублювання доменної логіки для невеликої команди.

## Рішення

Використовувати Next.js 14 App Router: Server Components для всіх читань, а Server Actions / Route Handlers — для всіх записів. Сторінки лежать під `app/`; мутації оформлені колокованими `'use server'` action-файлами (admin / auth / cabinet / shop). Мова визначається на сервері. Адмін-сторінки примусово динамічні (`force-dynamic`). Жодних прямих записів із браузера.

## Наслідки

- ✅ SSR за замовчуванням дає SEO та коректний перший рендер публічних даних без клієнтського JS для читань.
- ✅ Усі мутації проходять єдиним серверним шляхом (Server Actions / Route Handlers), де можна централізовано перевіряти сесію та роль.
- ✅ Action-файли колоковані поряд зі сторінками (`actions.ts` у кожному розділі), що тримає мутації близько до місця використання.
- ⚠️ Записи мають бути виключно серверними — будь-яка спроба зробити мутацію з клієнта є порушенням архітектури і вимагає окремого ревʼю.
- ⚠️ Адмін-розділ позначений `force-dynamic`, тож кешування там вимкнене — це навантаження на рендер потрібно враховувати при масштабуванні.
