# Як змінювати дизайн-систему

## Коли токен, а коли one-off

- Патерн повторюється **двічі+** або має повторюватись у подібному місці → **токен** у
  [styles.ts](../../components/ui/styles.ts)/[buttons.ts](../../components/ui/buttons.ts).
- Унікальний, разовий, нікуди не пошириться → можна локальний клас, але це **рідкість**.
  Якщо вагаєшся — токен.
- Перш ніж вписати клас руками — перевір [tokens-reference.md](tokens-reference.md).

## Коли потрібен ADR

Будь-яка зміна **правила** системи (новий тип кнопки, новий розмір шрифту, новий рівень
кольору, новий виняток, зміна ритму/шкали) фіксується [ADR](../architecture/adr/README.md)
у форматі MADR. Додавання токена, що лише виносить наявний хардкод **без зміни вигляду**,
ADR не потребує — досить запису в [tokens-reference.md](tokens-reference.md).

## Як додати виняток

Лише разом з ADR, що його обґрунтовує. Після прийняття — рядок у [05-exceptions.md](05-exceptions.md)
з посиланням на ADR. Винятків без ADR не буває.

## Чек-лист PR (UI)

- [ ] Жодних хардкод-класів кнопок/карток/заголовків/бейджів — лише `@/components/ui`.
- [ ] Жодного розміру шрифту поза `text-2xl/xl/base/sm/xs`.
- [ ] Жодного кольору-тексту поза токенами (`--c-brand-text`, сірі, семантичні `-fg`). Без `text-brand*`, `rose/emerald/amber-*`.
- [ ] Жодного `dark:` (крім FAB) — сірі токени перемикаються самі.
- [ ] Кнопки лише `btn("action")`/`btn("delete")`; бейджі лише `badgeClass()`.
- [ ] Заголовки/меню/кнопки — UPPERCASE; `cardTitle` — ні.
- [ ] Корінь сторінки — `ui.pageStack`; заголовок→контент — `mb-3`.
- [ ] Порожній стан — `ui.emptyState`; інлайн-лінк — `ui.link`.
- [ ] Той самий тип елемента в подібному місці виглядає **так само**, як уже є деінде.
- [ ] Жодного хардкод-рядка тексту в JSX — лише i18n (`st()`).
- [ ] `tsc --noEmit` і `next build` — зелені (перед пушем).

## Порядок при суперечці

Документ ↔ код розходяться → виправляємо так, щоб **код = документ = ADR**. Якщо документ
застарів — оновлюємо документ (і за потреби ADR), не «тихо» код.
