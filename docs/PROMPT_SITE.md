Продолжаем проект **RX Team** — сообщество ASG-игр (Wrocław).
Стек: **Next.js 14 (App Router)** + **grammY** (бот, webhook) + **Supabase**
(Postgres; secret key только на сервере, publishable key в браузере) + **luxon**
(Europe/Warsaw) + **Tailwind v3**. Деплой авто на push в `main` → `www.rxteam.pl`
(он же `rxteam.vercel.app`). Репо: github.com/ZeberdeeKh/Rxteam (приватный).
Полное ТЗ и статус — в **PLAN.md**: прочитай §15 (сайт), §16 (админка),
§19 (статус этапов), §21 (детальный план 6.1–6.4). Память/контекст проекта — там же.

## Что уже готово (Этап 6.0 — НЕ переделывать)
- Tailwind + лейаут (`app/layout.tsx`: хедер с навигацией/языком/логином, футер),
  нейтральный стиль (финальный дизайн будет Этап 7).
- i18n PL/EN/UA: `lib/site-i18n.ts` (`st(lang, key, vars)`, словарь UI),
  `lib/server-lang.ts` (`getServerLang()` из cookie `rx_lang` → Accept-Language → uk),
  `components/LangSwitcher.tsx`.
- Supabase-клиенты: `lib/supabase.ts` (service-key, привилегированные записи),
  `lib/supabase-server.ts` (SSR, сессия из cookie, `getAuthUser`, `isAuthConfigured`),
  `lib/supabase-browser.ts`. `middleware.ts` рефрешит сессию (fail-safe без env).
- Auth: **email+пароль** с подтверждением (`/register`, `/login`, `/auth/confirm`,
  `app/auth/actions.ts`: signUp/signIn/signOut) **и вход через Telegram Login
  Widget** (`/auth/telegram`, `lib/tg-auth.ts`, `lib/tg-session.ts` —
  своя подписанная cookie `rx_tg_session`, т.к. Supabase не знает TG-юзеров).
- Идентичность: таблицы `identities` + `link_codes` (`supabase/etap6.sql`),
  `lib/identities.ts`, резолвер `lib/session-player.ts` →
  `getSessionContext()` = `anon | unlinked | linked` (читает сперва TG-cookie,
  потом email-сессию). Линк email↔TG кодом из бота `/linksite`.
- Кабинет-заглушка `app/cabinet/page.tsx` (link-форма для unlinked / мини-профиль
  для linked).

## Архитектура — ЗАФИКСИРОВАНО, соблюдать:
- **RLS off.** Все записи/мутации — только через **Server Actions / Route
  Handlers** на сервере, с проверкой сессии и роли. Никаких прямых записей из
  браузера. Чтение публичных данных — тоже серверно.
- Маппинг сессии в игрока — ТОЛЬКО через `getSessionContext()`. Для проверки
  прав админа — поля `players.is_master`, `players.is_admin`, `players.admin_perms[]`
  (значения прав: `games|rental|checkin|referrals|players|joins`).
- Доменная логика переиспользуется из `lib/*` (НЕ дублировать): `lib/games.ts`
  (окна времени, `distanceMeters` haversine, `formatWhen`), `lib/economy.ts`,
  `lib/season.ts`, `lib/players.ts`, `lib/settings.ts` (`getSetting/setSetting/
  featureEnabled/getAllSettings`).
- i18n: ВСЕ новые тексты сайта — в `lib/site-i18n.ts` через `st()`, три языка
  PL/EN/UA. Никаких хардкод-строк в JSX.
- Tailwind, нейтральный стиль в духе уже существующих страниц (цвет `brand`).
- Новые поля/таблицы БД — отдельный файл `supabase/etap7.sql` (и далее), который
  организатор выполнит сам в Supabase SQL Editor. НЕ выполнять миграции за него.
- Бизнес-значения не выдумывать — заглушки в `settings`, спросить организатора.

## Дев-цикл (соблюдать на КАЖДОМ шаге)
`npx tsc --noEmit` → `npx next build` (должен быть зелёным) → commit →
`git push origin main` (деплой сам ~1 мин). Коммиты — осмысленными кусками
по под-этапам. Ветка `main`.

## ЗАДАЧА: реализовать 6.1 → 6.2 → 6.3 → 6.4 целиком, по шагам

Сначала задай организатору открытые бизнес-вопросы (ниже), затем реализуй по
порядку, с tsc+build+push после каждого логического куска.

### 6.1 Публичные страницы (доступны всем, без входа)
- Сначала вынеси общие серверные выборки в `lib/site-data.ts` (ближайшая игра,
  список игр, рейтинг) — переиспользуй с ботом где можно.
- `/` лендинг: герой + текст **складки PL** (дословно §15 PLAN.md, блок «Składka»),
  блок «ближайшая игра» (из `games` где `start_at>now`, ближайшая), ссылки.
- `/games`: ближайшая игра подробно + календарь/список будущих и прошедших игр
  (локация, дата/время через luxon Europe/Warsaw, число записавшихся).
- `/ranking`: таблица по «заработано всего» (как бот `/top`): позывной + ранг +
  заработано + сыграно. Пагинация (или топ-N + «показать все»). Публично.

### 6.2 Кабинет (после входа; `getSessionContext().state==='linked'`)
- Полный профиль: позывной, ранг, надёжность % (`lib/economy.ts`),
  заработано/баланс, сыграно, ачивки.
- История: `point_log` (дельты с причинами), сыгранные игры, ачивки.
- Запись/отписка на игру со страницы кабинета или игры (Server Action; те же
  правила, что в боте: рег до −9ч, отмена без штрафа до −24ч).
- **Самочек-ин для не-TG**: браузерная геолокация → Server Action → тот же
  `distanceMeters`+окно из `lib/games.ts`, запись `checkins.source='web'`.
- Решить с организатором: создавать ли «standalone» player для email-юзера,
  который НЕ привязал TG (сейчас он видит только форму привязки).

### 6.3 Магазин за баллы
- Миграция `supabase/etap7.sql`: `shop_items` (конфиг: id, title PL/EN/UA, cost,
  active, описание) + `purchases` (player_id, item_id, cost, at).
- Каталог + покупка за `points_balance` (Server Action: проверка баланса,
  списание, запись purchase, лог в `point_log`). Перечень бонусов — заглушка,
  наполнит организатор через админку.

### 6.4 Админка (только master/админы по правам; гейт на сервере)
- Доступ: `is_master` или нужное право в `admin_perms`. Не-админам — 404/redirect.
- Редактор `settings`: тоглы функций + числа (баллы, цены, стоимость рангов,
  тайминги, запас аренды, параметры лотереи) + тексты (онбординг/анонс),
  сгруппировано по разделам. Сохранение — Server Action через `setSetting`.
- Операционка: игры (создать/редактировать/отменить), регистрации, аренда
  (отметка занято/оплачено), чек-ины наживо, рефералы, игроки (баллы/штрафы/
  позывной), заявки в группу.
- Управление ролями админов (`is_admin`, `admin_perms[]`) — только master.
- **Экспорт CSV** (игроки, регистрации, чек-ины).
